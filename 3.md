# lo Resource-Oriented 重构

# Phase 3.5：Container 模型收敛技术实现文档

**目标：**

在 Phase 3 完成 Container Source 同步能力之后，对 Container 领域模型进行稳定化重构，消除生命周期状态、同步策略、数据来源之间的概念混淆，为后续 Resource Graph、Relation、Version 等上层能力提供稳定基础。

---

# 一、背景

## Phase 1 完成后的模型

Phase 1 将 lo 从 File-oriented 转换为 Resource-oriented。

核心变化：

```
文件
  ↓
Resource
```

升级为：

```
Resource Entity
    |
    +-- File Resource
    |
    +-- Container Resource
    |
    +-- Virtual Resource
```

文件只是 Resource 的一种存储形式。

---

## Phase 2 完成后的模型

Phase 2 引入 Container Member 生命周期。

原模型：

```
Container
    |
    |
    +-- 文件列表
```

升级：

```
Container Resource

    |
    |
    +-- Container Member

            |
            |
            +-- indexed
            +-- promoted
            +-- ignored
            +-- deleted
```

问题：

`status` 同时承担两个职责：

1. Member 生命周期
2. 同步参与状态

导致模型混合。

---

## Phase 3 完成后的模型

Phase 3 引入：

```
Resource Source
        |
        |
    Sync Engine
        |
        |
 Container Members
```

Container 成为外部数据源管理系统。

新增：

* source sync
* ignore matcher
* dirty tracking
* watcher integration

但是 Phase 3 暴露三个模型问题：

---

## 问题 1：ignored 不是生命周期状态

例如：

```text
文件:
secret.key


用户:
lo container ignore secret.key
```

它表达的是：

> 不参与同步

不是：

> 文件生命周期发生变化

而：

```
deleted
promoted
indexed
```

表达的是：

> 文件和 Resource 的生命周期关系

两个维度不应该放在同一个字段。

---

## 问题 2：Container Member 缺少 Source 归属

当前：

```
container_members

container_rid
path
hash
```

隐含假设：

> 一个 Container 只有一个 Source

但是未来：

```
Project Resource

    |
    +-- local folder
    |
    +-- git repository
    |
    +-- remote storage
```

多个 Source 可能存在。

如果两个 Source 都存在：

```
source A:

src/main.py


source B:

src/main.py
```

当前模型无法区分。

---

## 问题 3：同步配置和 Source 混合

当前：

```
resource_sources

resource_rid
location

enabled
sync_mode
last_scan_at
```

Source 描述：

> 数据在哪里

Sync 描述：

> 如何同步

两个概念不同。

---

# 二、Phase 3.5 目标

本阶段目标：

## 1. 分离 Member 生命周期和同步策略

从：

```
status

indexed
promoted
ignored
deleted
```

调整为：

```
status

indexed
promoted
deleted


force_ignore

0 / 1
```

---

## 2. 支持 Container 多 Source

增加：

```
source_id
```

使：

```
Container

    |
    +-- Source A
    |
    +-- Source B


Members

    |
    +-- source_id
```

---

## 3. 独立 Sync Policy

新增：

```
container_sync_configs
```

用于未来扩展：

* 自动同步
* 冲突策略
* 删除策略
* 调度策略

# 三、数据库设计变更

---

# 1. container_members 表调整

## 当前结构

```sql
container_members

id

container_rid

path

hash

status

resource_rid
```

---

## 修改后

```sql
CREATE TABLE container_members (

    id INTEGER PRIMARY KEY AUTOINCREMENT,

    container_rid TEXT NOT NULL,

    source_id INTEGER NOT NULL,


    path TEXT NOT NULL,

    hash TEXT,

    size INTEGER,

    mtime INTEGER,


    status TEXT NOT NULL DEFAULT 'indexed',

    resource_rid TEXT,


    force_ignore INTEGER DEFAULT 0,


    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,


    FOREIGN KEY(container_rid)
        REFERENCES resources(rid)
        ON DELETE CASCADE,


    FOREIGN KEY(source_id)
        REFERENCES resource_sources(id)
        ON DELETE CASCADE
);
```

---

# status 定义

## indexed

普通成员。

表示：

```
文件存在

被 Container 索引

没有独立 Resource
```

例如：

```
src/main.py
```

---

## promoted

已经提升为 Resource。

关系：

```
container_member

resource_rid=res_xxx
```

例如：

```
main.py

↓

Code Resource
```

---

## deleted

文件已经不存在。

但是保留：

```
resource_rid
```

用于恢复。

---

# 删除 ignored 状态

旧：

```
ignored
```

迁移：

```
status='indexed'
force_ignore=1
```

---

# 2. 新增 container_sync_configs

用于保存同步策略。

```sql
CREATE TABLE container_sync_configs (

    id INTEGER PRIMARY KEY AUTOINCREMENT,


    container_rid TEXT NOT NULL,

    source_id INTEGER NOT NULL,


    sync_mode TEXT DEFAULT 'manual',


    delete_policy TEXT DEFAULT 'soft',


    conflict_policy TEXT DEFAULT 'local',


    interval INTEGER,


    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,


    FOREIGN KEY(container_rid)
        REFERENCES resources(rid)
        ON DELETE CASCADE,


    FOREIGN KEY(source_id)
        REFERENCES resource_sources(id)
        ON DELETE CASCADE
);
```

---

字段说明：

## sync_mode

未来：

```
manual

auto

watch
```

---

## delete_policy

未来：

```
soft

hard

keep
```

---

## conflict_policy

未来：

```
local

remote

ask
```

---

# 3. resource_sources 保持职责单一

保持：

```sql
resource_sources

id

resource_rid

source_type

location

metadata
```

已有字段：

```
enabled
last_scan_at
```

可以保留。

但是：

删除：

```

```

迁移到：

```
container_sync_configs
```

# 四、状态模型调整

## 修改前

```
                 promote

indexed ---------------- promoted

   |
   |
 ignore

   |

```

---

## 修改后

生命周期：

```
                    promote

indexed -------------------- promoted

   ^                           |
   |                           |
   |                           |
   +--------- demote -----------

             delete


deleted -------------------- indexed/promoted

             restore
```

同步策略：

独立：

```
force_ignore


0

|

1
```

# 十、Phase 3.5 完成标准

完成后：

* Container Member 生命周期单一职责
* ignore 不再污染状态机
* Container 支持多 Source
* Source 和 Sync Policy 分离
* 数据模型可以支持未来：

  * Git Source
  * Remote Source
  * 自动同步
  * 冲突解决
  * 多仓库资源聚合

Phase 3.5 的核心不是增加功能，而是**冻结 Container 领域模型，避免后续 Resource Graph 和 Version Layer 建立在不稳定的数据结构上。**
