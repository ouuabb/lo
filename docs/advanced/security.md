## 安全设计摘要

### 安全措施总览

| 安全措施 | 说明 |
|---------|------|
| 密钥权限 0o600 | 仅所有者可读写 |
| GCM 认证标签 | 自动检测文件篡改 |
| 随机 IV（每次加密不同）| 相同明文产生不同密文 |
| close() 内存擦除 | 密钥不留内存残留 |
| getter 返回 Buffer 副本 | 防引用被意外清零 |
| backup 排除 .repo/keys/ | 密钥不随备份传播 |
| HKDF Salt + Info 绑定 | 防预计算和密钥重用 |
| execFileSync | 防命令注入 |
| 挑战 nonce 随机化 | 防重放攻击 |

### 分层安全模型

lo 采用多层防护，每层独立负责：

| 层次 | 机制 | 防护目标 |
|------|------|---------|
| 传输层 | SSH/SCP 加密通道 | 网络窃听 |
| 传输层 | HTTP API Bearer Token + SSH 挑战-应答 | API 未授权访问 |
| 文件层 | LOEC 加密格式（AES-256-GCM）| 中继服务器数据泄露 |
| 密钥层 | RepoKey + HKDF + protected_*.key | 硬盘被盗/丢失 |
| 完整性层 | 每 batch 内含 SHA-256 校验 | 数据传输损坏/篡改 |

### 密钥安全详解

**密钥权限：**

`.repo/keys/` 下的文件权限设为 `0o600`（仅所有者可读写）。任何读取操作都会先检查权限。

**内存安全：**

- `RepoKey` 只存在于内存中，不写入磁盘（绑定 SSH 后）
- 使用 `fill(0)` 在 `close()` 时擦除内存中的密钥
- 返回密钥时给出 `Buffer` 副本，防止原始引用被意外修改

**GCM 认证标签：**

AES-256-GCM 的认证标签（128-bit）保证：
- 任何对密文的篡改都会被检测到
- 解密前先验证标签，不通过则拒绝解密
- 防止攻击者构造恶意密文尝试注入

**随机 IV：**

每次加密使用不同的 96-bit 随机初始化向量（IV），确保即使相同内容多次加密也产生不同密文。这防止了攻击者通过对比密文来推断内容相似性。

**HKDF 安全：**

- **Salt**：32 字节安全随机盐值，防止预计算攻击（彩虹表）
- **Info** 上下文绑定（"lo-repo-key-protection-v1"），防止密钥重用
- 将非均匀分布的 SSH 私钥内容转化为均匀分布的 256-bit 密钥

### 传输安全

**SSH/SCP：**

- `lo push/pull/clone` 底层使用系统 `scp` 命令
- 传输通道由 SSH 协议保护（密钥交换 + 对称加密 + HMAC 认证）
- 使用系统 SSH 配置（`~/.ssh/config`），可配置代理跳板、端口转发

**HTTP API：**

- 默认只监听 `127.0.0.1`，不暴露到网络
- 使用 SSH 挑战-应答认证（私钥不离开客户端）
- session token 有效期 60 分钟
- nonce 一次性使用，防重放攻击

### 服务器安全

中继服务器被攻破后，攻击者能获取：

| 可获取 | 不可获取 |
|--------|---------|
| batch 文件名（含时间戳）| 文件内容（加密）|
| manifest.json（设备 ID、操作数量）| 加密密钥 |
| ops.json（操作类型、RID、路径、散列）| 明文笔记 |
| 文件名（原始文件名 + LOEC 后缀）| — |

> 如果你介意文件名泄露，可以在 lo 中不使用包含敏感信息的文件名。

### 服务器 SSH 加固建议

```bash
# /etc/ssh/sshd_config
PasswordAuthentication no
PermitRootLogin prohibit-password
PubkeyAuthentication yes
```

### 安全威胁模型

**防范的场景：**

- ✅ 硬盘/存储介质被盗或丢失（有 lo auth 时）
- ✅ 中继服务器被攻破
- ✅ 备份文件泄露
- ✅ 网络传输被窃听
- ✅ 文件被篡改
- ✅ API 重放攻击

**无法防范的场景：**

- ❌ 攻击者已登录你的账户（RepoKey 在内存中时）
- ❌ 键盘记录器或屏幕录制
- ❌ 恶意软件截获内存中的密钥
- ❌ 物理接触已解锁的设备

> 这些是所有本地加密方案的共同局限，不是 lo 特有的问题。

### 最佳实践

1. **始终使用 `lo auth add`** — 即使单机使用也建议绑定 SSH 密钥
2. **离线备份 RepoKey** — 打印到纸上或存入密码管理器
3. **备份排除密钥** — `lo backup` 自动排除 `.repo/keys/`
4. **服务器 SSH 加固** — 禁用密码登录，只允许密钥认证
5. **定期测试恢复** — 验证你确实能从备份恢复数据
6. **不要分享 repo.key** — RepoKey 是保护笔记的唯一凭证
7. **环境变量 LO_AUTH_SKIP=1** — 仅用于 CI/CD 环境
