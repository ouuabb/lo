Resource Repository 架构设计（V1）

一、设计目标

本系统不是一个 Markdown 笔记管理器，也不是一个 文件管理器。

它是一个 Resource Repository（资源仓库）。

整个系统只有一个核心概念：

一切皆 Resource。

Markdown、PDF、图片、视频、网页、音频……本质上没有区别，它们都是 Resource，只是拥有不同的类型（Type）。

因此，系统不存在"附件"的概念，也不存在"笔记拥有文件"的概念。

只有：

Resource 与 Resource 之间建立关系（Relation）。

整个系统围绕 Resource Graph（资源图）运行。

二、核心思想

系统由两个基本对象组成：

Resource Relation 

除此之外，其它概念都只是它们的扩展。

例如：

Markdown ↓ Resource(type=note) PDF ↓ Resource(type=pdf) Image ↓ Resource(type=image) 

未来增加任何资源类型，都无需修改架构，只需要新增一个 Type。

三、Repository

整个 Workspace 就是一个 Repository。

例如：

workspace/ notes/ resources/ exports/ .repo/ database.sqlite config.json thumbnails/ indexes/ 

注意：

真正的 Repository 不是文件夹。

真正的 Repository 是：

.repo/database.sqlite 

文件只是 Repository 管理的数据。

数据库才是整个系统的核心。

四、Resource

每一个资源都有唯一身份。

例如：

Resource RID Type Path Hash Metadata Created Updated 

各字段职责：

RID

资源永久身份。

一旦生成，永不改变。

整个系统所有引用都引用 RID。

不是 Path。

不是文件名。

不是 Hash。

Type

资源类型。

例如：

note pdf image video audio html folder ... 

Path

当前资源位置。

可以修改。

移动文件时：

Path 更新。

RID 不变。

Hash

资源当前内容指纹。

内容变化：

Hash 更新。

RID 不变。

Hash 用于：

判断内容是否变化

去重

检测同步

快速比较

它不是资源身份。

Metadata

资源元数据。

例如：

Title Author Width Height Pages Size Mime ... 

不同 Type 可以拥有不同 Metadata。

五、RID（Resource ID）

RID 是整个系统唯一身份。

例如：

01K3... 或者 res_xxxxx 

RID 在资源进入 Repository 时生成。

以后：

修改内容 移动 重命名 修改 Metadata 

RID 永远不变。

所有 Relation 都引用 RID。

六、Relation

Resource 之间不存在特殊关系。

全部统一表示为 Relation。

例如：

from to type 

例如：

Markdown ↓ Image type = embed Project ↓ PDF type = contain Book ↓ Author type = reference 

所有关系都是统一的数据结构。

七、Resource Graph

整个 Repository 实际是一张图。

Markdown │ ├──── PDF │ ├──── Image │ └──── Audio 

Graph 中：

节点：

Resource 

边：

Relation 

系统真正维护的是这张 Graph。

八、文件系统

磁盘只是 Resource 的存储介质。

Repository 不依赖目录结构。

目录只是 View。

例如：

Project A 

不是：

Folder 

而是：

Resource ↓ contain ↓ Resource 

以后完全可以：

同一个 Resource

属于多个 Project。

九、Repository 索引

Repository 维护资源索引。

例如：

RID ↓ Path ↓ Hash ↓ Metadata 

任何查询：

都来自索引。

而不是扫描磁盘。

例如：

图片墙：

Type = image 

即可。

十、文件监听

Repository 持续监听 Workspace。

例如：

Add Delete Rename Change 

收到事件：

更新数据库。

例如：

Path Hash Metadata 

保持 Repository 与磁盘一致。

Repository 永远认为：

数据库是真相。

磁盘是数据源。

十一、资源生命周期

资源进入系统：

Import ↓ 生成 RID ↓ 建立索引 ↓ 进入 Graph 

资源修改：

更新 Hash 更新 Metadata 

资源移动：

更新 Path 

资源删除：

标记删除 或 真正删除 

整个生命周期都围绕同一个 RID。

十二、查询（Query）

系统所有能力，本质都是查询。

例如：

图片墙：

Type = image 

PDF 库：

Type = pdf 

所有 Markdown：

Type = note 

未引用资源：

没有任何 Relation 指向 

最近导入：

Created DESC 

某资源引用了哪些资源：

Relation from = RID 

哪些资源引用了它：

Relation to = RID 

整个系统无需针对图片、PDF、Markdown 分别开发功能。

只需要开发：

查询。

十三、静态输出（View）

任何静态页面都是 Repository 的一个 View。

例如：

Gallery Library Timeline Homepage Map Tag Tree 

全部来自 Query。

例如：

SELECT ↓ 生成 HTML 

Markdown 并不是唯一输出对象。

任何 Resource 都可以生成自己的 View。

十四、CLI 定位

CLI 不负责业务。

CLI 只是 Repository 的操作入口。

例如：

resource create resource import resource remove resource move resource link resource unlink resource search resource export 

而：

note create pdf import image add 

都只是这些命令的快捷方式。

CLI 永远操作 Repository。

不会直接操作文件。

十五、整个架构

CLI │ ▼ Repository API │ ┌───────────────┴───────────────┐ ▼ ▼ Resource Service Relation Service │ │ └───────────────┬───────────────┘ ▼ Repository Database │ ┌────────────┴────────────┐ ▼ ▼ File Watcher Query Engine │ │ └────────────┬────────────┘ ▼ File System 

十六、设计原则

整个系统遵循以下原则：

一切皆 Resource：不存在特殊对象，Markdown、PDF、图片等都只是不同类型的 Resource。

身份与内容分离：资源身份由稳定的 RID 表示；内容变化通过 Hash 反映，两者互不替代。

关系是一等公民：系统真正维护的是 Resource Graph，而不是目录树。所有引用、包含、派生等联系都通过 Relation 表达。

Repository 为中心：数据库负责索引、关系和元数据，磁盘负责存放文件，两者各司其职。

视图来源于查询：图片墙、文献库、时间线、项目视图等都不是独立功能，而是针对 Resource 与 Relation 的不同查询结果。

类型可扩展：新增一种资源类型，不需要修改架构，只需要定义新的 Type 及其元数据解析方式。

按照这个设计，系统的演化方向就不会受限于"Markdown 笔记软件"，而是可以逐渐发展为一个面向知识资源的 Repository 与 Resource Graph 平台。
